## User Scenarios & Testing *(mandatory)*

### User Story 1 - コマンド実行による客観的条件の検証 (Priority: P1)

ユーザーはタスクの完了条件として「テストが全パスする」「カバレッジが90%以上」などの客観的・数値的条件を、シェルコマンドの終了コードで検証できるようにしたい。これにより、LLM の誤判定や揺らぎに依存せず、再現可能な判定が可能になる。

**Why this priority**: LLM による数値判定の不確実性はこの機能の最大の動機であり、コマンド検証がなければ機能全体の価値がない。

**Independent Test**: コマンド型の条件のみを持つタスクを実行し、コマンドの終了コードに基づいて正しく met/not met が判定されることを確認できる。

**Acceptance Scenarios**:

1. **Given** ユーザーがコマンド型の完了条件（例: テスト実行コマンド）を指定したタスクを投入した, **When** 判定フェーズでそのコマンドが終了コード 0 を返した, **Then** その条件は「met」と判定される
2. **Given** ユーザーがコマンド型の完了条件を指定したタスクを投入した, **When** 判定フェーズでそのコマンドが非ゼロの終了コードを返した, **Then** その条件は「not met」と判定され、コマンドの出力が判定結果の根拠として記録される
3. **Given** コマンド型の条件が met と判定された, **When** 他のすべての条件も met である, **Then** タスクは完了と判定される

---

### User Story 2 - 意味的条件とコマンド条件の混在 (Priority: P1)

ユーザーは1つのタスクに対して、意味的条件（LLM 判定）とコマンド条件（終了コード判定）を混在させて指定できる。これにより、主観的な品質基準と客観的な数値基準を組み合わせた柔軟なタスク定義が可能になる。

**Why this priority**: 実際のタスクでは主観的条件（「コードが読みやすい」）と客観的条件（「テスト通過」）が混在するのが一般的であり、この組み合わせが実用上必須。

**Independent Test**: 意味的条件とコマンド条件を混在させたタスクを実行し、各条件が適切な方式で判定されることを確認できる。

**Acceptance Scenarios**:

1. **Given** 意味的条件とコマンド条件が混在するタスクを投入した, **When** 判定フェーズに到達した, **Then** コマンド条件はコマンド実行で判定され、意味的条件は LLM で判定される
2. **Given** コマンド条件はすべて met だが意味的条件が not met の場合, **When** 判定が行われた, **Then** タスクは未完了と判定される
3. **Given** 意味的条件はすべて met だがコマンド条件が not met の場合, **When** 判定が行われた, **Then** タスクは未完了と判定される

---

### User Story 3 - コマンド実行エラーの明示的な報告 (Priority: P2)

コマンドが正常に完了しなかった場合（タイムアウト、コマンド未検出など）、システムはエラーを明示的に報告する。エラーを隠蔽して処理を続行することはない。

**Why this priority**: 設定ミスや環境の問題を早期に検出し、無意味なイテレーションの繰り返しを防ぐ。

**Independent Test**: 存在しないコマンドやタイムアウトするコマンドを条件として指定し、エラーが適切に報告されることを確認できる。

**Acceptance Scenarios**:

1. **Given** シェル自体の起動に失敗するコマンド条件を指定した（OSError）, **When** 判定フェーズでコマンドを実行しようとした, **Then** 実行エラーとしてループが停止し、エラー内容がユーザーに報告される
2. **Given** タイムアウト時間を超えるコマンドをコマンド条件として指定した, **When** 判定フェーズでコマンドがタイムアウトした, **Then** タイムアウトエラーとしてループが停止し、エラー内容がユーザーに報告される
3. **Given** 存在しないコマンド名をコマンド条件として指定した（例: typo）, **When** 判定フェーズでシェルが終了コード 127 を返した, **Then** 条件は「not met」として正常処理される（FR-009 準拠。ループは停止しない）

---

### Edge Cases

- コマンド条件のコマンドが空文字の場合はどうなるか？ → 入力バリデーションで拒否する
- コマンド条件のみ（意味的条件なし）のタスクの場合、LLM 判定エージェントはどう振る舞うか？ → コマンド結果のみで判定が完結し、LLM は呼び出されない
- コマンドが非常に大量の出力を生成した場合は？ → 根拠として記録する出力は 10KB に制限する
- 複数のコマンド条件がある場合の実行順序は？ → 定義順に順次実行する
- コマンドの終了コードが 0 でも stderr に出力がある場合は？ → 終了コード 0 = met として扱い、stderr は根拠として記録する

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは完了条件として「意味的条件」と「コマンド条件」の2種類を受け付けなければならない
- **FR-002**: 意味的条件は親仕様（001-endless8-engine-core）に従い LLM により判定されなければならない
- **FR-003**: コマンド条件はシェルコマンドの終了コードにより判定されなければならない（終了コード 0 = met、非ゼロ = not met）
- **FR-004**: 完了条件が文字列のみで指定された場合、意味的条件として扱わなければならない
- **FR-005**: コマンド条件は各イテレーションの判定フェーズで実行されなければならない（実行エージェント完了後、LLM 判定前）
- **FR-006**: コマンドの実行結果（終了コード、標準出力、標準エラー出力）は条件判定結果の根拠に含めなければならない
- **FR-007**: コマンド条件の判定結果は、意味的条件の LLM 判定時にコンテキストとして提供されなければならない（親仕様 FR-008 を拡張し、判定エージェントは ExecutionSummary に加えてコマンド実行結果を追加の情報源として受け取る）
- **FR-008**: コマンドの実行にはタイムアウト制限を設けなければならない
- **FR-009**: コマンドの実行エラーは明示的なエラーとしてループを即座に停止しなければならない（not met として処理を続行してはならない。複数のコマンド条件がある場合、最初のエラーで停止し残りは実行しない）。実行エラーとは、コマンドプロセスの起動失敗（コマンド未検出、権限不足）およびタイムアウトを指す。コマンドプロセスが正常に起動し終了コードを返した場合は、終了コードの値に関わらず実行エラーとはみなさない
- **FR-010**: コマンド条件のみのタスクでは、LLM 判定エージェントの呼び出しを省略できなければならない
- **FR-011**: コマンド条件の判定結果の確信度は確定的（揺らぎなし）でなければならない
- **FR-012**: コマンド条件には人間向けの説明（表示名）を付与できなければならない
- **FR-013**: 各条件の判定結果には、どの判定方式（意味的/コマンド）で判定されたかを含めなければならない
- **FR-014**: コマンドはプロジェクトディレクトリを作業ディレクトリとして実行されなければならない
- **FR-015**: コマンド条件の実行と結果、および各条件の判定結果（根拠・判定方式を含む）は履歴ログに記録されなければならない

### Key Entities

- **完了条件（Criterion）**: タスクの完了を判定するための個別の条件。判定方式（意味的/コマンド）、条件テキストまたはコマンド、人間向け説明を持つ
- **条件判定結果（Criteria Evaluation）**: 個別の完了条件に対する判定結果。条件の met/not met 状態、エビデンス、確信度、判定方式を持つ
- **コマンド実行結果（Command Result）**: コマンド条件の実行結果。終了コード、標準出力、標準エラー出力、実行時間を持つ

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: コマンド型の完了条件が指定されたタスクにおいて、判定の再現性が100%（同一のコマンド・同一の環境状態に対して常に同一の判定結果）
- **SC-002**: 文字列のみの条件定義で作成されたタスクが正常に動作する
- **SC-003**: コマンドの実行エラー（タイムアウト、コマンド未検出）が発生した場合、100%の確率でユーザーに明示的に報告される（エラーの隠蔽が0件）
- **SC-004**: 意味的条件とコマンド条件を混在させたタスクで、各条件が正しい判定方式で評価される（誤った方式での判定が0件）

## Clarifications

### Session 2026-03-05

- Q: コマンド出力の記録サイズ上限は？ → A: 10KB に制限する
- Q: 複数コマンド条件で最初にエラーが発生した場合の挙動は？ → A: 最初のエラーで即座に停止し、残りのコマンドは実行しない

## Assumptions

- コマンド条件はユーザーが明示的に指定するものであり、サンドボックスによる制限は不要（実行エージェントの allowed_tools 制約とは独立）
- コマンド条件の実行はイテレーションごとの判定フェーズで行う（実行エージェント → サマリ → コマンド検証 → LLM 判定の順）
- コマンド条件の判定結果を意味的条件の LLM 判定時にコンテキストとして渡す（LLM がコマンド結果を参照して判定品質を向上）
- コマンド結果は ExecutionSummary には含めない（ExecutionSummary は実行エージェントの作業記録であり、判定フェーズの結果は別管理）
- コマンド条件の確信度は常に 1.0（コマンドの判定は確定的）
- コマンド条件は定義順に順次実行する（並列実行は行わない）
- FR-007 による親仕様 FR-008 の拡張（判定エージェントの情報源追加）は、plan フェーズで親仕様との優先順位を整理する

## Scope Boundaries

### In Scope

- 意味的条件（semantic）とコマンド条件（command）の2種類の完了条件
- コマンドの終了コードによる met/not met 判定
- 文字列のみの条件指定（意味的条件として扱う）
- コマンド実行のタイムアウト制限
- コマンド実行エラーの明示的報告
- 判定結果への判定方式の記録

### Out of Scope

- 条件間の依存関係（AND 以外の論理演算）
- コマンド出力値を次の条件に渡すパイプライン
- 条件の動的生成（LLM が新しい条件を追加する）
- リモート実行（SSH 先でのコマンド実行）
- コマンド条件の並列実行
